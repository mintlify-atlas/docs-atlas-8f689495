The Dashlane Public API exposes two endpoints for password health data: a current snapshot of team-wide metrics and a historical record of how those metrics have changed over time. You can use these endpoints to build dashboards, set up security alerts, and demonstrate measurable improvement in your team's password hygiene.

## The PasswordHealthReport endpoint

`PasswordHealthReport` returns the current state of password health across your entire team. The response includes aggregate counts and a per-member breakdown, so you can identify which individuals need attention.

```bash
curl --request POST \
  --url https://api.dashlane.com/public/teams/PasswordHealthReport \
  --header 'Authorization: Bearer DLP_teamUuid_accessKey_secretKey' \
  --header 'Content-Type: application/json' \
  --data '{}'
```

A realistic response:

```json
{
  "report": {
    "passwordsCount": 842,
    "safePasswords": 701,
    "weakPasswords": 54,
    "reusedPasswords": 72,
    "compromisedPasswords": 15,
    "averagePasswordStrength": 76.4,
    "members": [
      {
        "login": "alice@example.com",
        "passwordsCount": 120,
        "safePasswords": 115,
        "weakPasswords": 2,
        "reusedPasswords": 3,
        "compromisedPasswords": 0,
        "averagePasswordStrength": 88.1
      },
      {
        "login": "bob@example.com",
        "passwordsCount": 95,
        "safePasswords": 60,
        "weakPasswords": 18,
        "reusedPasswords": 14,
        "compromisedPasswords": 3,
        "averagePasswordStrength": 52.3
      }
    ]
  }
}
```

### Understanding the fields

<AccordionGroup>
  <Accordion title="passwordsCount">
    The total number of passwords stored across all members. This is the denominator used to interpret the category counts below.
  </Accordion>
  <Accordion title="safePasswords">
    Passwords that meet Dashlane's strength criteria and have not appeared in known data breaches. Aim for this number to approach `passwordsCount`.
  </Accordion>
  <Accordion title="weakPasswords">
    Passwords that fail Dashlane's strength evaluation — typically short, simple, or dictionary-based. These are flagged regardless of breach exposure.
  </Accordion>
  <Accordion title="reusedPasswords">
    Passwords shared across two or more accounts for the same member. Reuse amplifies the impact of any single credential compromise.
  </Accordion>
  <Accordion title="compromisedPasswords">
    Passwords that have appeared in known data breach databases. These require immediate attention regardless of their apparent strength.
  </Accordion>
  <Accordion title="averagePasswordStrength">
    A score from 0 to 100 reflecting the mean strength of all passwords in scope. Scores above 80 are generally considered strong.
  </Accordion>
</AccordionGroup>

<Note>
  Password health data is only available for members who have the Dashlane desktop app installed and have synced their vault. Members with `hasDesktopApp: false` will not appear in the per-member breakdown. See [team management](/guides/team-management) for how to check desktop app enrollment.
</Note>

## The PasswordHealthHistory endpoint

`PasswordHealthHistory` returns dated snapshots of the same metrics, giving you a time series to track progress or detect regressions.

```bash
curl --request POST \
  --url https://api.dashlane.com/public/teams/PasswordHealthHistory \
  --header 'Authorization: Bearer DLP_teamUuid_accessKey_secretKey' \
  --header 'Content-Type: application/json' \
  --data '{}'
```

A realistic response with dated snapshots:

```json
{
  "history": [
    {
      "date": "2024-01-01",
      "passwordsCount": 780,
      "safePasswords": 610,
      "weakPasswords": 88,
      "reusedPasswords": 65,
      "compromisedPasswords": 17,
      "averagePasswordStrength": 68.2
    },
    {
      "date": "2024-02-01",
      "passwordsCount": 805,
      "safePasswords": 651,
      "weakPasswords": 74,
      "reusedPasswords": 59,
      "compromisedPasswords": 21,
      "averagePasswordStrength": 71.0
    },
    {
      "date": "2024-03-01",
      "passwordsCount": 842,
      "safePasswords": 701,
      "weakPasswords": 54,
      "reusedPasswords": 72,
      "compromisedPasswords": 15,
      "averagePasswordStrength": 76.4
    }
  ]
}
```

Use the historical data to:

- Plot `averagePasswordStrength` over time to show security improvement to leadership
- Detect sudden increases in `compromisedPasswords` that could indicate a breach event
- Track whether remediation campaigns (e.g., after a phishing incident) had a measurable effect

## Use case: threshold-based alerts

A practical pattern is to poll `PasswordHealthReport` on a schedule and trigger an alert when key metrics exceed acceptable thresholds.

The following script uses `jq` to evaluate thresholds and exit with a non-zero status if any are breached, making it compatible with standard monitoring pipelines:

```bash
#!/usr/bin/env bash

API_KEY="${DASHLANE_API_KEY}"
MAX_COMPROMISED=0
MAX_WEAK_PERCENT=10

REPORT=$(curl --silent --request POST \
  --url https://api.dashlane.com/public/teams/PasswordHealthReport \
  --header "Authorization: Bearer ${API_KEY}" \
  --header 'Content-Type: application/json' \
  --data '{}')

COMPROMISED=$(echo "$REPORT" | jq '.report.compromisedPasswords')
TOTAL=$(echo "$REPORT" | jq '.report.passwordsCount')
WEAK=$(echo "$REPORT" | jq '.report.weakPasswords')

# Alert if any compromised passwords exist
if [ "$COMPROMISED" -gt "$MAX_COMPROMISED" ]; then
  echo "ALERT: ${COMPROMISED} compromised passwords detected"
  exit 1
fi

# Alert if weak password percentage exceeds threshold
WEAK_PERCENT=$(echo "scale=1; $WEAK * 100 / $TOTAL" | bc)
if (( $(echo "$WEAK_PERCENT > $MAX_WEAK_PERCENT" | bc -l) )); then
  echo "ALERT: Weak passwords at ${WEAK_PERCENT}% (threshold: ${MAX_WEAK_PERCENT}%)"
  exit 1
fi

echo "Password health OK: ${COMPROMISED} compromised, ${WEAK_PERCENT}% weak"
```

<Warning>
  Never hardcode your API key in scripts. The example above reads the key from the `DASHLANE_API_KEY` environment variable. See [API key management](/authentication/api-keys) for rotation and scoping guidance.
</Warning>

## Best practices for polling

<Tip>
  Call `PasswordHealthHistory` at most once daily — the snapshots are date-granular and more frequent calls will return the same data. For `PasswordHealthReport`, an hourly cadence is sufficient for most monitoring use cases. See [automation patterns](/guides/automation) for recommended cron schedules.
</Tip>

Consider the following polling intervals based on your use case:

| Use case | Endpoint | Recommended interval |
|---|---|---|
| Security dashboard | `PasswordHealthReport` | Every 1–4 hours |
| Trend visualization | `PasswordHealthHistory` | Once daily |
| Incident alerting | `PasswordHealthReport` | Every 30–60 minutes |
| Executive reporting | `PasswordHealthHistory` | Once weekly |
