When a request fails, the Dashlane Public API returns an HTTP status code in the 4xx or 5xx range alongside a JSON body that describes the problem. Use the status code to understand the category of failure, and the error body to identify the specific cause.

## Error response format

All error responses follow a consistent structure:

```json
{
  "error": {
    "code": "UNAUTHORIZED",
    "message": "The provided API key is invalid or has expired."
  }
}
```

The `code` field is a machine-readable identifier suitable for conditional error handling in your integration code. The `message` field provides a human-readable explanation.

## Error codes

<AccordionGroup>
  <Accordion title="400 — Bad Request">
    The server could not process the request because the request body is malformed or contains invalid parameters.

    **Common causes:**
    - The request body is not valid JSON
    - A required parameter is missing from the body
    - A parameter value is the wrong type (e.g., a string where a number is expected)

    **How to resolve:**
    - Verify your request body is valid JSON and includes the `Content-Type: application/json` header
    - Check the endpoint reference page to confirm which parameters are required and their expected types
    - Use a JSON linter or validator to catch syntax errors before sending

    ```bash
    # Correct: valid JSON body
    curl --request POST \
      --url https://api.dashlane.com/public/teams/Members \
      --header 'Content-Type: application/json' \
      --header 'Authorization: Bearer DLP_teamUuid_accessKey_secretKey' \
      --data '{}'
    ```
  </Accordion>

  <Accordion title="401 — Unauthorized">
    The request did not include a valid API key, or the API key has expired or been revoked.

    **Common causes:**
    - The `Authorization` header is missing from the request
    - The API key format is incorrect — it must follow `Bearer DLP_<teamUuid>_<accessKey>_<secretKey>`
    - The API key has been revoked in the Dashlane admin console
    - The access key or secret key contains a typo or extra whitespace

    **How to resolve:**
    - Confirm the `Authorization` header is present and uses the `Bearer` prefix
    - Double-check the full key value, including all three segments separated by underscores
    - Generate a new API key in the Dashlane admin console if the existing one has been revoked
    - See [managing API keys](/authentication/managing-keys) for instructions on rotating keys

    <Warning>
      Never log or print your full API key in application output or error messages. If a key is accidentally exposed, revoke it immediately and generate a new one.
    </Warning>
  </Accordion>

  <Accordion title="403 — Forbidden">
    The API key is valid, but it does not have permission to access the requested resource.

    **Common causes:**
    - The team UUID in the API key does not match the team being queried
    - The API key belongs to a member account without admin access to the requested data

    **How to resolve:**
    - Verify the team UUID in your API key matches the team you intend to query
    - Confirm the key was generated by an account with the appropriate admin role
    - Check the [authentication overview](/authentication/overview) for information on required permissions
  </Accordion>

  <Accordion title="404 — Not Found">
    The requested endpoint path does not exist.

    **Common causes:**
    - A typo in the endpoint URL path
    - Calling an endpoint that does not exist in the API

    **How to resolve:**
    - Compare your URL against the [endpoint list](/api-reference/overview#available-endpoints) to confirm the path is correct
    - Verify you are using the correct base URL: `https://api.dashlane.com`
  </Accordion>

  <Accordion title="429 — Too Many Requests">
    Your application has exceeded the API rate limit and requests are being throttled.

    **Common causes:**
    - Sending requests too frequently in a short time window
    - Running multiple concurrent processes that all call the API simultaneously

    **How to resolve:**
    - Check the `Retry-After` response header, which indicates how many seconds to wait before retrying
    - Implement exponential backoff: wait progressively longer between retries (e.g., 1 s, 2 s, 4 s, 8 s)
    - Reduce polling frequency by caching responses where the underlying data changes infrequently
    - If you consistently hit rate limits at normal usage levels, contact Dashlane support to discuss your use case

    <Tip>
      For automation scripts, add a short delay between API calls and avoid running the same script in parallel across multiple processes. Even a 1–2 second pause between requests is usually enough to stay within the rate limit.
    </Tip>
  </Accordion>

  <Accordion title="500 — Internal Server Error">
    An unexpected error occurred on the Dashlane server side. This is not caused by your request.

    **Common causes:**
    - A transient infrastructure issue on Dashlane's side
    - A rare edge case in request processing

    **How to resolve:**
    - Wait a moment and retry the request — transient errors often resolve on their own
    - If you receive repeated 500 errors across multiple retries and endpoints, contact Dashlane support

    <Note>
      Isolated 500 errors are usually transient. Build retry logic with a short delay into your integration so that occasional server errors do not cause your automation to fail permanently.
    </Note>
  </Accordion>
</AccordionGroup>

## Auditing for accidentally committed API keys

Dashlane API keys start with the prefix `DLP_`. If you are concerned that a key may have been accidentally committed to version control or included in application logs, search your codebase and repository history for this prefix.

<Tip>
  Run a search for `DLP_` across your repository before each release to catch any accidentally hardcoded keys. Use environment variables or a secrets manager to store API keys instead of committing them to source code.
</Tip>

If you discover a key has been exposed, revoke it immediately in the Dashlane admin console and generate a replacement. See [managing API keys](/authentication/managing-keys) for step-by-step instructions.
